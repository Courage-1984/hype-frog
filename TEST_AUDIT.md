# hype-frog Test Suite Audit

Generated by a full pass over `tests/` (123 files, 807 test functions, 15 subdirectories) cross-referenced against every source package in `src/hype_frog/`. Intended as a working checklist for adding tests, upgrading weak ones, and consolidating overlap ‚Äî not a finished report to file away.

Every claim below carries a file (and line, where useful) so it can be re-verified quickly before acting on it. A few claims from the first automated pass were spot-checked against source and one was corrected (noted inline) ‚Äî treat this document the same way: re-check anything load-bearing before you build on it, since the codebase moves fast.

## 1. Executive summary

- **123 test files, 807 test functions**, spread across 15 subdirectories that mirror `src/hype_frog/` package-for-package (per `tests/CLAUDE.md`).
- **No coverage tooling at all** ‚Äî no `pytest-cov`, no `.coveragerc`, no `[tool.coverage.*]`, no threshold, no CI workflow in the repo. Gaps below were found by manual/agent cross-referencing, not a coverage report ‚Äî running one for real would be a fast, cheap first step (see ¬ß6).
- **Naming is 100% consistent** (`test_*.py`), **zero skip/xfail markers** anywhere, and all 123 files contain substantive test bodies (no empty stubs).
- **rules/ and extractors/ are the strongest packages** in the codebase: all 99 `IssueRule` entries in `rules/registry.py` are individually parametrized and positive/negative tested, with a completeness-guard test that fails CI if a new rule ships without a trigger case. `extractors/` has proportionally the best ratio of tests to public functions of any package.
- **The weakest spots cluster in `orchestration/` (Excel/report assembly), a handful of `crawler/`/`core/` utility modules, and PDF content validation.** The single biggest structural gap is `orchestration/export_workbook.py` (771 lines, assembles ~35 Excel sheets) having no dedicated test file ‚Äî it's only exercised indirectly, and only for sheet *names*, never cell content of the real assembled output.
- One test-infra inconsistency: `tests/CLAUDE.md` documents an `@pytest.mark.integration` policy for `tests/integration/`, but the marker isn't registered in `pytest.ini` and isn't applied to the one file that lives there ‚Äî it silently runs in the default suite.

## 3. Critical gaps ‚Äî P0 (live, non-trivial code with no real coverage)

1. **`orchestration/export_workbook.py`** (771 lines) ‚Äî `write_full_suite_workbook()` assembles ~35 Excel sheets and has **no dedicated test file**. It's only exercised indirectly by 3 tests in `tests/orchestration/test_export_flow.py`, and those assert **sheet names only** (`"Main" in workbook.sheetnames`, `REQUIRED_FULL_SUITE_SHEETS.issubset(...)`) ‚Äî never a single cell value from the real, assembled workbook. Cell-level assertions exist elsewhere (`test_executive_dashboard.py`, etc.) but only against hand-built in-memory workbooks calling sheet-builder functions directly, not the real end-to-end pipeline output. **This is the single biggest gap in the codebase.**
2. **`orchestration/export_executive_reports.py`** ‚Äî the PDF/HTML orchestration glue is never exercised, even indirectly. All 3 tests in `tests/orchestration/test_export_flow.py` (lines 122-123, 179-180, 227-228) explicitly `monkeypatch.delenv("HF_EXPORT_PDF")` / `delenv("HF_EXPORT_HTML")` before running, so the code path that wires a real crawl+export into PDF/HTML generation is unit-tested for its downstream pieces (`test_pdf_exporter.py`, `test_html_report_*.py`) but never through this glue layer.
3. **`crawler/psi_cache.py`** ‚Äî the on-disk SQLite PSI response cache (`cache_get`, `cache_put`, `open_cache_db`, `CACHE_TTL_SECONDS`) has **zero test references anywhere**, and isn't re-exported through the `psi_engine` facade that other PSI tests go through.
4. **`diagnostics/quick_test.py`** ‚Äî has a same-named test file, `tests/core/test_quick_test.py`, which creates a false sense of coverage: that file actually imports and tests `core/run_config.py::quick_test_run_config`, not anything in `diagnostics/quick_test.py`. The real gate driver (`run_quick_test_gate`, `_preflight_phase`, `_audit_phase`, `_validate_crawl_rows`, `_print_report`, `run_quick_test_pytest`) has **zero direct tests**.
5. **`core/env_vars.py`** ‚Äî 47 accessor functions, only ~5-7 are directly asserted anywhere (mostly `get_hf_gsc_url_inspection`, `get_hf_max_memory_mb`, and a couple set-but-not-asserted flags via `test_run_setup.py`). All PDF/HTML branding getters, theme getters, snapshot-retention/db-path getters, and formatting-disable flags are untested.
6. **`reporter/stream_workbook.py`** and **`reporter/engine_io.py`** ‚Äî no direct test file for either; both are only touched transitively when the full-suite export path runs inside `test_export_flow.py`. The streaming sheet-writer and formula-injection helpers in `engine_io.py` (16 public defs) have no targeted assertions.
7. **`core/numeric_utils.py`** ‚Äî `safe_float`, `round2`, `round4`, `clamp_pct` are untested despite being used pervasively across pipeline/reporter for numeric coercion. Only `safe_int` gets indirect exercise (via `tests/analysis/test_delta_engine.py`).
8. **`core/console.py`** (`log_phase_banner`, `log_stage_timer`, `log_startup_panel`) and **`core/path_bootstrap.py`** (`bootstrap_src_path`, `repo_root`, `src_root`, `package_root`) ‚Äî zero test coverage.
9. **`crawler/client.py::create_session`** ‚Äî this is **live, actively used code** (imported through `hype_frog/crawler/__init__.py`, called in `crawl_runner.py`'s BFS loop), not dead code. Its real behavior ‚Äî the User-Agent header and `TCPConnector` limit/keepalive settings it constructs ‚Äî is never exercised; every test that touches the crawl loop monkeypatches `create_session` away entirely (`tests/orchestration/test_crawl_runner.py`).
10. **`pipeline/context.py::PipelineContext`** ‚Äî genuinely dead code. Confirmed zero importers anywhere in `src/` (only its own definition references itself). Worth confirming intent and deleting rather than writing tests for it.
11. **PDF export content** ‚Äî `tests/reporter/test_pdf_exporter.py` only asserts `Path(pdf_path).exists()` and `.stat().st_size > 0`. No `pypdf`/`pdfplumber`/`fitz` import exists anywhere in the suite, so nothing verifies that KPI values, branding, or layout actually render correctly in the PDF ‚Äî this is the one output format that's purely "did it not crash."

## 4. Lackluster / weak tests ‚Äî P1

1. **`tests/orchestration/test_regen_report.py:64-85`** (`test_regen_report_skips_crawl_and_calls_export`) ‚Äî `execute_export`, `execute_crawl`, `run_enrichment`, and `save_crawl_snapshot` are **all four** patched out. The test only asserts which mocks were/weren't called (`crawl.assert_not_called()`, `export.assert_called_once()`, etc.) ‚Äî it never verifies what `execute_export` actually does with the loaded snapshot. Pure control-flow test, not a behavior test of `--regen-report`.
2. **`tests/orchestration/test_content_planner.py:196`** ‚Äî `assert len(rows) >= 0` is a tautology (`len()` can never be negative). The test name (`test_root_url_with_trailing_slash_matches_normalized_key`) promises a real normalization check that the body doesn't deliver.
3. **Facade-blur in PSI and delta tests** ‚Äî `tests/crawler/test_psi_engine.py` (17 tests) exercises `psi_batch.py` and `psi_merge.py` only indirectly through the `psi_engine` re-export facade; `tests/analysis/test_delta_engine.py` (7 tests) does the same for `delta_loader.py`, `delta_models.py` (22 Pydantic-model public defs, ~2 tests touch it), and `delta_sheet_builder.py`. It's hard to tell from the test file alone which underlying file/function is actually being covered.
4. **`crawler/gsc_engine.py`** ‚Äî 24 public defs, only 2 are directly tested (`tests/crawler/test_gsc_auth_paths.py`); the rest of the GSC data-shaping logic is only exercised as a side effect of enrichment/export-flow tests.
5. **`orchestration/crawl_runner_bfs.py`, `crawl_runner_frontier.py`, `crawl_runner_interactive.py`** ‚Äî no dedicated test files; all three are only exercised indirectly through `crawl_runner.py`'s public API in `test_crawl_runner.py`.
6. **`orchestration/export_row_builders.py`** ‚Äî 14 public defs, no direct tests; `build_aeo_rows`/`build_aioseo_rows` are imported into `test_export_flow.py` via the `app_orchestrator` re-export rather than from `export_row_builders` itself, which further obscures what's actually under test.
7. **Unseeded randomness**: `tests/crawler/test_network_engine.py:76-80` (`test_poisson_jitter_positive_is_finite_positive`) calls `_poisson_jitter_seconds(0.5)` 50 times with no fixed seed, asserting only loose bounds (`> 0.0`, not NaN/inf). Low practical flake risk given the loose bounds, but it's real unseeded randomness in a test.
8. **`diagnostics/full_smoke_test.py`** ‚Äî `tests/core/test_full_smoke_test.py` only imports and tests `_validate_full_smoke_rows` (1 of 6 public defs). The actual driver (`_apply_full_smoke_env`, `_preflight_phase`, `run_full_smoke_pytest`, `_print_report`) is untested, mirroring the `quick_test.py` gap above.
9. **`config_defaults.py`** ‚Äî 21 public defs (mostly default dicts/constants), only exercised as a side effect of `tests/config/test_config_loader.py`'s merge/override tests ‚Äî no per-constant assertions.

## 5. Mock & hardcoded-data audit

**Mocking is mostly legitimate** ‚Äî the project's own convention (`tests/CLAUDE.md`: "mock `aiohttp` sessions and Playwright with `unittest.mock`; real network only in `tests/integration/`") is followed consistently. 32 files use `unittest.mock`/`monkeypatch`; the large majority mock only the I/O boundary (HTTP session, Playwright page, spaCy import) while exercising the real function/class under test. Heaviest mock users: `test_network_engine.py` (~63 sites, mocks Playwright + `asyncio.sleep`), `test_api_clients.py` (~36, mocks `aiohttp` responses), `test_crawl_runner.py` (~54, stubs `fetch_and_parse`/`create_session`/`AuditCache`/`IntentAnalyzer` via `monkeypatch.setattr` ‚Äî broad seams, but the BFS scheduling logic itself runs for real).

**The one exception** is `test_regen_report.py` (¬ß4.1), which mocks out the unit under test's collaborators so completely that nothing real is verified.

**Hardcoded inline data** is present but modest ‚Äî all inline HTML/dict blobs found are small (under ~20 lines each): `test_data_assembler.py` (9 blocks), `test_nav_footer_links.py` (11 blocks), `test_heading_outline.py` (8 blocks), plus single small blocks in `test_og_social.py`, `test_fetcher_extraction.py`, `test_competitor_benchmarks.py`. This is a mild positive versus large pasted fixture blobs.

**Worth flagging**: `src/hype_frog/diagnostics/full_smoke_fixtures.py` is production code (not a test file) that generates synthetic URLs/rows programmatically (status/extraction-state cycling via modulo). A large share of "integration-looking" tests (`test_export_flow.py`, `test_enrichment_flow.py`, the full-smoke-test suite) run against this synthetic generator rather than real crawled data or file-based fixtures ‚Äî worth remembering when judging how much those tests actually prove about real-world crawl output.

**Golden-file/fixture-based testing** is limited to `tests/integration/test_sample_page_pipeline.py`, which reads real static fixtures from `tests/fixtures/` (`sample_page.html`, `sample_schema.json`, `sample_psi_response.json`, plus `empty_page.html`, `hreflang_cluster.html`, `malformed_schema.html`, `aeo_content.html` exposed via `tests/conftest.py` fixtures). All fixture files are small (under ~2.3 KB). There is no true snapshot/diff-style test anywhere (comparing a full generated Excel/PDF/HTML report byte-for-byte against a saved golden file) ‚Äî Excel output is checked via targeted `openpyxl` cell lookups instead, which is generally the better pattern but doesn't exist for the real full-suite workbook (see ¬ß3.1).

## 6. Test infrastructure gaps

- **No coverage tooling** ‚Äî no `pytest-cov`, no `.coveragerc`, no `[tool.coverage.*]` in `pyproject.toml`, no threshold. Cheapest first move: add `pytest-cov` and run `uv run pytest --cov=hype_frog --cov-report=term-missing` once to get a real, current number instead of relying on manual cross-referencing like this document.
- **Marker/policy mismatch**: `pytest.ini` registers only `slow` and `network` markers (`--strict-markers` is on). `tests/CLAUDE.md` states integration tests "must be marked `@pytest.mark.integration`... excluded from the default run," but `integration` isn't registered, and the sole file in `tests/integration/` (`test_sample_page_pipeline.py`) carries no marker at all ‚Äî it silently runs as part of `uv run pytest` today, contradicting the documented policy. Either register the marker and apply it, or update the doc to match reality.
- **No CI workflow** found in the repo (no `.github/workflows/`) ‚Äî the quick-test/full-smoke-test gates (`uv run hype-frog --quick-test` / `--full-smoke-test`) are the closest thing to a gate, and both are run manually.
- Zero `@pytest.mark.skip`/`xfail` anywhere ‚Äî not a gap, just noted as a clean baseline to preserve.

## 7. Consolidation opportunities ‚Äî P2

1. **`reporter/` has 32+2 files**, several of which are phase-numbered regression-guard files that likely overlap in what they guard: `test_phase2_info_travel.py`, `test_phase3_polish.py`, `test_phase3_ux_polish.py`, `test_ux_overhaul.py`. Before adding more to this pile, do a pass to check for duplicate assertions (e.g. multiple files re-asserting TOC sync or nav-strip presence) and consider merging into fewer, topic-named files (e.g. `test_toc_and_navigation.py`, `test_workflow_status.py`) rather than sprint/phase-named ones, which will keep accumulating.
2. **Facade test files should mirror source files, or be documented as intentionally facade-level.** `test_psi_engine.py` and `test_delta_engine.py` currently blur which of several underlying source files (`psi_batch.py`/`psi_merge.py`, `delta_loader.py`/`delta_models.py`/`delta_sheet_builder.py`) is actually being exercised by a given test. Either split into per-file test modules matching the `tests/` mirrors-`src/` convention, or add a short comment at the top of each explaining the intentional facade-level scope.
3. **`export_row_builders.py`** tests are imported via the `app_orchestrator` re-export rather than the source module directly (¬ß4.6) ‚Äî retarget those imports to the real module so coverage tools and future readers can tell what's actually under test.
