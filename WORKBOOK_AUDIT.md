## 0. Run scope — read this first

**The crawl was seeded from `page-sitemap.xml`, not the site's real sitemap index.** The very first line of the crawl log reads `Fetching sitemap from: https://africanmarketingconfederation.org/page-sitemap.xml` → `Found 87 URLs across 1 sitemap file(s)`. The site's actual `/sitemap.xml` is a sitemap *index* with 10 child files — three post sitemaps, a product sitemap, a category sitemap, three tag sitemaps, and the page sitemap that was actually used. None of the others were seeded.

That exact URL, `page-sitemap.xml`, is hard-coded in `src/hype_frog/core/run_config.py:19` as `_DEFAULT_SMOKE_SITEMAP_URL` — the fixture target this codebase's own `--quick-test`/`--full-smoke-test` gates use, on this same domain. This is **not a code bug**: the sitemap-index recursion logic in `src/hype_frog/crawler/sitemap.py` (handles `sitemapindex` root tags and recurses into child sitemaps) looks correctly implemented. Whatever launched this run simply pointed it at the page sitemap directly rather than the site root.

Practically, this means blog posts, products, categories, and tags were excluded from all 33 sheets — the 255 URLs audited are pages only, not the whole site. Worth re-running against `https://africanmarketingconfederation.org/sitemap.xml` if full-site coverage was the intent.

## 2. High severity

Bugs that materially distort scores or counts used across multiple sheets — these change what a reader would conclude about the site.

### H1 — SEO Health Score / Technical Health / SEO Score pegged near zero for almost the entire site

**Sheets:** Main, Technical Diagnostics, Priority URLs, Executive Briefing

`score_url_health()` computes `100 − 25×Critical − 10×Warning − min(10, 3×Observation)`, floored at 0. Verified: 254 of 255 measured URLs score exactly **0** (the 255th is null/"Unmeasured"). Three near-universal critical rules do it alone — "Lab LCP Above 4.0s (Mobile)" (253/255), "Broken Internal Links" (246/255), "Near-Duplicate Content" (153/255, itself a downstream artifact of H2 below). `Technical Health` is mathematically derived from this same score (`compute_seo_technical_copy_scores`), so it is also 0 for all 255 rows, and `SEO Score = 0.5×Technical Health + 0.25×Copy Score + 0.25×(something)` silently collapses to exactly `0.25 × Copy Score` — confirmed arithmetically across sampled rows with zero residual (e.g. Copy Score 77.83 → SEO Score 19.46, exact, across 120 rows).

Every sheet that ranks or scores by these fields loses its ability to distinguish "bad" pages from "worse" ones. This is a **scoring-formula design concern**, not a wiring bug — the math is applied correctly everywhere it was checked, but the penalty weights (25/Critical, 10/Warning) combined with how many near-universal critical rules exist on this site mean the score has zero discriminating power for this crawl.

### H2 — Sidebar/widget content bleeds into extracted page body, homogenizing entity & AEO scores across ~half the site

**Sheets:** Content Optimisation Hub, Main

This WordPress/Elementor site has no `<main>`/`<article>` tags on most templates, so the extractor's "primary content region" logic falls back to `content_soup.body`, stripping only `nav, header, footer, aside, script`. A shared "recent posts" sidebar widget survives that strip and gets folded into the analyzed body text on completely unrelated pages — homepage, a 2021 press release, and a 2026 conference page all return the identical entity list `"Kenyan | Cannes Lions Grand Prix | Paid Sick Leave"`.

Confirmed sitewide:
- `Semantic AEO Score` (col AK) = exactly `60` for 191/247 rows (77%)
- `Entity Density (%)` (col AH) = exactly `88.24` for 118/247 rows
- `Top Entities` (col AI) identical across 119/246 non-empty rows (48%)
- The same shared widget's `<h3>` heading ("Kenyan agency wins first Cannes Lions Grand Prix in nation's history") appears as literal duplicated H3 content on 120/168 rows

This is the most plausible driver of the false 153/255 "Near-Duplicate Content" hit rate feeding H1 above, and passes through into `Content Hub Metrics!AEO Visibility Gain` (`100 − aeo_score`; 191/247 rows = exactly `40`, matching the 60-score cluster).

Compounding factor: **spaCy isn't installed for this run** (`Semantic engine using keyword fallback for entity density / Top Entities / Semantic AEO Score` — logged warning), so entity extraction used the coarser keyword fallback rather than real NER. Running `uv sync --extra semantic` (or `uv run hype-frog --install-semantic`) before the next production run is a cheap, separate improvement worth making regardless of the content-region fix.

### H3 — Link Intelligence!H (Broken Internal Links Count) is off-by-one for every one of 255 rows

**Sheet:** Link Intelligence (hidden)

`apply_link_intelligence_summary_broken_formulas` writes a self-referencing `SUMPRODUCT` formula into column H for every "Summary"-type row, keyed to the loop's own row number, via `link_inventory_broken_per_source_formula(f"$A{row}")`. This runs **before** `add_return_to_briefing_strip` inserts a banner row at row 1 via `worksheet.insert_rows(1)`. openpyxl's `insert_rows` shifts cell positions but does **not** rewrite formula text, so every previously-injected `$A{row}` self-reference now points one row too high.

Verified empirically against all 255 Summary rows (manually recomputing "broken internal instances per source URL" straight from Link Inventory): **0/255** rows match their own URL's correct value; **255/255** exactly match the row-above's value instead. Concrete example: `Link Intelligence!H3` (homepage) formula references `$A2`, which after the row-insert is the header row (literal text "URL"), so it never matches anything → cached value **0**. The homepage actually has **2** broken internal outbound links. Most other rows only look correct by coincidence — one sitewide dead nav link (`amc-conference-2025-delegate-packages`) contributes 2 broken instances to nearly every page regardless of the shift, masking the bug for 222/255 rows.

The debug-level log noise seen during export (`"Count CF skipped at Link Intelligence!H<row>..."`) is a **separate, benign** issue — `apply_generic_sheet_coloring` tries `int(val or 0)` on a cell that now holds formula text, catches the `ValueError`, and just skips a static color fill. It is not the cause of the wrong count, just a symptom that a formula (not a literal) sits there.

The hidden helper column **"Broken Links (computed)"** (col U), which holds a literal Python-computed value written *before* the row-insert, is unaffected by the shift and matched manual counts for 252/255 rows (3 minor mismatches not further root-caused).

- `src/hype_frog/reporter/engine_io.py:75-109` (formula injection)
- `src/hype_frog/reporter/sheets/navigation.py:69-98` (`insert_rows(1)`, causes the shift)
- `src/hype_frog/pipeline/broken_links.py:92-113` (`link_inventory_broken_per_source_formula`)

### M6 — "Probable duplicate" rows sometimes name no counterpart, giving nonsensical redirect advice

9 of 126 "Duplicate Content" rows in Template & Duplication Risks (e.g. `amc-forum-mombasa`, `amc-awards-2024`, `publications-2`, `amc-conference-2022`, `book-now`) show `Issue = "Probable draft or near-duplicate page (100% content similarity)"` with a **blank** "Example URLs" target column, yet still carry the generic "Exact Action" instruction to "remove or noindex the draft/copy page, 301 redirect to the primary page" — nonsensical when applied to the page that IS the primary.

Root cause: `probable_duplicate = bool(duplicate_of) or draft_flag or in_heading_cluster` — a page can be `in_heading_cluster=True` (matched against a genuine duplicate elsewhere) yet win the `_should_point_to_duplicate` rank comparison and keep `duplicate_of=None`, so it's flagged with a populated similarity score but no named counterpart. The sheet builder only ever prints `duplicate_of`, never the raw `best_match_url` actually used to compute the similarity number. Verified against companion rows: `amc-marketing-conference-2024` correctly names `amc-forum-mombasa` as its duplicate-of target (100% similarity); `amc-forum-mombasa`'s own row shows the identical 100% similarity claim with a blank target.

The underlying detection logic — heading-fingerprint clustering + Jaccard token similarity — looks sound, and the pairs it *does* name are plausible (e.g. `-copy`/`-test` slug siblings at 84%/94% similarity).

- `src/hype_frog/pipeline/content_duplicates.py:122-123,243`
- `src/hype_frog/reporter/sheets/merged_builders.py:819-847` (`build_template_duplication_risks_rows`)
